# 东南亚本地支付 — 代付退款需求（Payout Link 方案）

## 1. 背景

QRIS、PromptPay、PayNow、DuitNow、VietQR 这些走 Payloco 收单的东南亚本地支付退不回原路，PayKKa 现在干脆藏起退款按钮，商户遇客诉没法退。

本需求做「代付退款」：钱不原路退，改打到消费者的收款账户。核心是让消费者自己填卡号——系统生成收款链接（Payout Link）发给消费者，商户全程不碰账户信息（竞品 Xendit 同思路）。

**整体一条线**：

```
发起退款（只填金额）→ 运营先审核该不该退
  → 通过 → 发收款链接给消费者 → 消费者自填卡号/钱包号
  → PayKKa 打款（支持的自动打，不支持的人工打）
  → 成功 SUCCESS ｜ 失败 FAILURE（打款失败 / 链接过期 / 放弃等，商户可再退）
```

---

## 2. 功能描述（MPS / API / OPS）

发起、审核两步跟现在的退款完全一样（只填金额、审「该不该退」）。审核通过后先路由判定 refundStrategy，再分岔到原路退 / 代付退两条独立路径。

### 2.1 场景故事说明

**场景：曼谷商户小王，PromptPay 退款 1,500 THB**

**人物**

- **小王**：曼谷一家跨境电商商户，用 MPS 后台管订单
- **阿明**：在泰国用 PromptPay 付了 1,500 THB 买耳机，收到货后申请退货退款
- **小李**：PayKKa OPS 运营，负责审核、代付
- **支付方式**：PromptPay（走 Payloco 收单，不能原路退回）

**① 发起 + 审核（共用段）**

阿明联系小王要退款。小王打开 MPS 交易详情，点「退款」。

系统先查可退金额，退款按钮可用。小王只填 1,500 THB 和支付密码，提交后退款单 RG21100xxxx，状态 审核中（REVIEWING）。小王在退款详情里看到「等运营审核」。

次日小李在 OPS 退款审核里看到这笔单：金额合理、无重复退款、非恶意客诉，点通过。

**② 路由：代付退（不是原路退）**

审核通过后，系统判定走 PAYOUT_LINK 代付退。PayKKa 生成 Payout Link，按下单预留手机号给阿明发短信。

**③ 消费者填链接**

阿明点开链接，看到退款金额 1,500 THB 和 PromptPay 收款字段，填自己的银行账号，勾选「仅用于本次退款」，提交。

系统内部进入 PAYING（打款中），对外仍是 PROCESSING。链接一次性失效。

**④ 打款：三种可能（故事分叉）**

**路径 A · 自动打款（AUTO）**

若该商户 + PromptPay 支持自动代付：PayKKa 调 payout/create，钱由 Payloco 打到阿明账户。几分钟后 webhook 回来，状态 SUCCESS。阿明收到退款，客诉结束。

**路径 B · 人工 · 商户打款**

若 PromptPay 不能自动代付，且 PayKKa 与该商户的协议约定：退款由商户自行打出。

小李在 OPS 代付退款列表看到单子「打款中」。小李通知小王：「请按信息线下转 1,500 THB 给阿明。」小王从自己公司账户转给阿明 PromptPay 绑定账户。转完后小李在 OPS 点确认商户已打款，状态 SUCCESS。

**路径 C · 人工 · 渠道/银行代付**

同样是 PromptPay 不能自动代付，但 PayKKa 与合作银行签的是：由银行代付。

小李在 OPS 代付退款列表看到单子「打款中」。小李联系银行：银行把 1,500 THB 打到阿明账户，回执给小李。小李在 OPS 确认回执，状态 SUCCESS。

## 3. 退款状态

两套：退款 4 态（保持原样）；内部 5 态（仅 OPS 菜单可见）。

### 3.1 退款状态（保持原样）— 4 个

| status | 商户看到 | 什么时候 |
|--------|----------|----------|
| REVIEWING | 审核中 | 发起后等审核 |
| PROCESSING | 处理中 | 审核通过 → 成功/失败之前 |
| SUCCESS | 退款成功 | 打款完成 |
| FAILURE | 退款失败 | 审核拒绝、运营放弃、打款失败、链接过期 |

### 3.2 内部状态 — 5 个

| internal_status | OPS 看到 | 什么时候 | 对外映射 |
|-----------------|----------|----------|----------|
| REVIEWING | 待审核 | 发起后 | REVIEWING |
| WAITING_FILL | 待消费者填链接 | 代付退审核通过，链接已发且未过期 | PROCESSING |
| PAYING | 打款中 | 消费者填完；或原路退审核通过 | PROCESSING |
| SUCCESS | 退款成功 | 钱打出去了 | SUCCESS |
| FAILURE | 退款失败 | 打款失败 / 链接过期 | FAILURE |

WAITING_FILL / PAYING 对外一律 PROCESSING。LINK_EXPIRED 直接 FAILURE。

## 4. 路由（退款方式判定）

退款方式不用商户选，系统自己判断。

### 4.1 三种退款方式

| # | 退款方式 | 什么时候走 | 卡号谁填 | 谁打款 |
|---|----------|------------|----------|--------|
| ① | 原路退（老功能） | 支持原路退的 | 不用填 | 原路退回 |
| ② | 代付退（收款链接） | 退不回原路，但可代付 | 消费者自己填 | 自动 or 人工 |
| ③ | 不可退 | 上面都不满足 | — | 按钮置灰 |

## 8. 收款页

🖥 UI：https://uii-wheat.vercel.app/payout/AbCd1234

消费者点开链接填收款账户，字段由「收款字段模板」配置。

### 8.5 链接规则

| 项 | 规则 |
|----|------|
| 有效期 | 可配置（如 7 天）；超时未填 → 定时任务置 FAILURE，推 webhook |
| 一次性 | 提交成功 → PAYING，页面失效 |
| 过期后 | 本笔 FAILURE；商户再退一笔拿新链接 |
